# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

A web app that generates short Japanese stories constrained to the kanji a user has previously seen (read in another story) plus a JLPT-N5 baseline, using OpenRouter LLM APIs. Designed for Japanese reading practice via comprehensible input. Each user has their own kanji exposure history, story history, and per-word LLM Q&A threads.

## Commands

```bash
# Install dependencies
npm install

# Start local Supabase (requires Docker)
npx supabase start

# Apply migrations and seed data (kanji + dev test user)
npx supabase db reset

# Generate seed SQL from data/kanji.json (only needed if kanji data changes)
npm run generate-seed

# Convert a JPDB Yomitan freq dict zip ‚Üí client/public/frequency/jpdb.json
# (only needed when bumping the dict). Source: github.com/Kuuuube/yomitan-dictionaries
npm run generate-frequency -- path/to/JPDB_v2.2_Frequency_YYYY-MM-DD.zip

# Join the surface-keyed JPDB index with jmdict-simplified to produce
# client/public/frequency/jpdb-by-entry.json (per-JMdict-entry rank). Run
# after `generate-frequency` whenever either JPDB or JMdict is bumped.
# Source: github.com/scriptin/jmdict-simplified ‚Äî pass the jmdict-eng-*.json.tgz
# from the latest release.
npm run generate-frequency-by-entry -- path/to/jmdict-eng-X.Y.Z+yyyymmdd.json.tgz

# Sync prod Supabase data into local instance
npm run sync-prod

# Generate TypeScript types from Supabase schema
npx supabase gen types typescript --local > client/src/lib/database.types.ts

# Run client dev server
npm run dev

# Build client (typecheck + Vite build)
npm run build

# Lint client
npm run lint --workspace=client

# Test client (Vitest)
npm test                              # one-shot (fast `unit` project)
npm run test:watch --workspace=client # watch mode

# Word-index regression suite ‚Äî runs the real detection pipeline against the
# vendored JMdict snapshot (see "Word-index regression testing" below)
npm run test:index
npm run test:index:accept             # bless reviewed detection changes into baselines

# Force-refresh the JMdict snapshot in client/src/test/jpdict/ (a gitignored
# build artifact; the word-index suite auto-records it on first run otherwise)
npm run record-test-dictionary

# Serve Edge Functions locally
npx supabase functions serve --env-file .env.local

# Deploy Edge Functions
npx supabase functions deploy generate-story
npx supabase functions deploy translate-sentence
npx supabase functions deploy openrouter-usage
```

## Architecture

This is an npm workspaces monorepo. The root `package.json` declares `client` as the only workspace; there is no `server/` directory ‚Äî the backend is entirely Supabase (Postgres + Edge Functions).

### Client (`client/`) ‚Äî Vite + React + TypeScript

Talks directly to Supabase (DB via SDK, Edge Functions via `functions.invoke`). Routing in `client/src/App.tsx`: `/` redirects to `/stories`; lazy-loads `/stories`, `/stories/:id`, `/stats`, `/settings`; everything except `/login` is wrapped in `<ProtectedRoute>` + `<AppLayout>`. Story generation is a modal on the Compositions page (`/stories`), not a standalone route. Plain CSS colocated as `Component.css` next to `Component.tsx` ‚Äî no Tailwind, no CSS modules. ### Word-index regression testing

Locks in word-detection behaviour (`regroupWords` / `extractWordOccurrences`) and tracks accuracy as the algorithm improves. The detection pipeline depends on JMdict (`@birchill/jpdict-idb`, IndexedDB-backed), kuromoji, and the JPDB frequency JSON ‚Äî all browser-shaped. `client/src/test/headlessDictionary.ts` boots the *genuine* stack in Node: `fake-indexeddb` + a `self` shim run jpdict-idb headless against a JMdict snapshot in `client/src/test/jpdict/` ‚Äî a **gitignored build artifact** the harness auto-records from the CDN on first run (`record-test-dictionary.mjs`; also runnable directly). jpdict-idb's `update()` has no public `baseUrl`, so the snapshot is replayed through a `fetch` stub that also serves the frequency JSON; kuromoji reads its dict via `VITE_KUROMOJI_DICT_PATH`. `npm run test:index` therefore exercises the real algorithm offline and reproducibly.

Workflow: curate a story's index in the app with the override editor, click **Export test fixture** on its detail page, drop the `<slug>.json` into `client/src/test/fixtures/word-index/`. `wordIndex.fixtures.test.ts` runs the real `extractWordOccurrences` on each fixture's `content`, and `diffWordIndex` (`lib/wordIndexFixture.ts`) compares it three ways: against the curated `expected` index (each span flagged `manual` ‚Äî true = a hand-fix, i.e. a known algorithm gap) and against the machine-managed `<slug>.baseline.json`. A run fails when the algorithm breaks a non-`manual` span (a `regression`) or when anything changed since the baseline; `npm run test:index:accept` blesses reviewed changes into the baselines (it cannot mask a regression ‚Äî that's checked against `expected`, not the baseline). Manual spans the algorithm catches up on surface as `improvement`s. See `client/src/test/fixtures/word-index/README.md`.

### Supabase (`supabase/`)

- `config.toml` ‚Äî local CLI config; `db.seed.sql_paths` runs both `seed.sql` and `seed_dev.sql` on `db reset`.
- `seed.sql` ‚Äî joyo kanji reference data (regenerated from `data/kanji.json` via `npm run generate-seed`).
- `seed_dev.sql` ‚Äî dev test user (`dev@local.test` / `devpassword`) and sample stories. Idempotent on re-runs. Still inserts grade 1‚Äì3 rows into `user_kanji` for historical parity, but those rows are inert ‚Äî no code path reads `user_kanji.known` anymore.
- `migrations/` ‚Äî timestamped SQL files (`YYYYMMDDHHMMSS_*.sql`). Run `ls supabase/migrations/` for current state.
- `functions/_shared/` ‚Äî utilities shared by Edge Functions (`openrouter.ts`, `story.ts`, `text.ts`, `log.ts`). `log.ts` is a dependency-free `logOpenRouter(event, fields, failed?)` ‚Äî every OpenRouter call (streaming + non-streaming + the usage proxy) emits a one-line structured `[openrouter] {json}` record covering request, result, and the raw upstream body on any non-OK status, so a failed background generation is diagnosable from the Edge Function logs alone. `failed` records route to `console.error` so the Supabase log viewer flags them.
- `functions/generate-story/` ‚Äî runs as a **background task**. Auths user, rejects (409) if a `status='generating'` row already exists for the user, inserts a placeholder `stories` row with `status='generating'`, returns 202 + `{ story_id }` immediately, then uses `EdgeRuntime.waitUntil` to call OpenRouter, parse the title/content, compute difficulty (loads kanji metadata server-side), and `UPDATE` the row to `status='complete'` (or `status='failed'` + `error_message` on any error). The client never streams from this function ‚Äî it polls `stories.id` until status flips. Pins to a single model via `ALLOWED_MODELS` allow-list (see `generate-story/index.ts:24`). Maps 401/402/429 to user-friendly errors.
- `functions/translate-sentence/` ‚Äî translates a single sentence within a story to natural English. Input is `{ story_id, sentence_start, sentence_end, regenerate? }` ‚Äî sentence offsets are char offsets in the *cleaned* content (ruby-stripped) as computed by the client's `extractSentenceSnippet`, so both sides agree on sentence bounds. The prompt sends the **full cleaned passage** alongside the target sentence so the model can resolve dropped subjects, pronouns, names, and tone consistency ‚Äî Japanese drops a lot, and a context-free translation often guesses wrong. On cache hit (and `regenerate=false`) returns the stored translation without calling OpenRouter; otherwise calls the model, upserts `stories.translations[<start>-<end>] = { text, model, generated_at }`, and returns the translation. Pins a single model (see `translate-sentence/index.ts`).
- `functions/openrouter-usage/` ‚Äî surfaces OpenRouter credit/usage to the client.

## Data Model

- `kanji` ‚Äî reference data (read-only). `character` PK, `grade` (1-6, 8=secondary), `jlpt` (5=easiest, 1=hardest, NULL allowed), `meanings`, `readings_on`, `readings_kun`
- `user_kanji` ‚Äî legacy per-user known state. Kept in the schema for safety but no longer read or written by any client code or RPC. "Seen kanji" is now derived purely from read stories (see `KanjiContext` / `getKanjiExposures`); a future migration may drop this table.
- `stories` ‚Äî per-user stories. Columns include `title`, `content`, `paragraphs` (INT NOT NULL ‚Äî legacy; fixed at 3 for every story, no longer user-selectable or surfaced in the UI; the `generate-story` Edge Function writes a constant), `topic`, `formality` (`impolite`/`casual`/`polite`/`keigo`), `content_type` (`fiction`/`nonfiction`), `filters` JSONB (legacy, no longer written by the client), `allowed_kanji` (TEXT ‚Äî concatenated kanji string, not JSON), `difficulty` JSONB, `translations` JSONB nullable (`{ "<sentence_start>-<sentence_end>": { text, model, generated_at } }` ‚Äî a sentence-level cache populated by the `translate-sentence` Edge Function), `read_count` (INT NOT NULL, defaults 0; weights kanji-exposure aggregation so re-reads count), `first_read_at` / `last_read_at`, `word_index_at` TIMESTAMPTZ nullable (stamped by `index_story_words` when the story's regrouped spans are bulk-uploaded into `story_word_occurrences`; null means the indexer hasn't run yet and the next backfill pass will trigger it), `word_index_version` INTEGER nullable (stamped alongside `word_index_at` as a provenance record of which `WORD_INDEX_VERSION` in `lib/storyWordIndex.ts` indexed the story ‚Äî **not** a re-index trigger: bumping the constant does not re-index existing stories, the backfill (`getStoriesNeedingIndex`) only picks up stories whose `word_index_at` is null. To re-index already-stamped stories after a pipeline change, ship a one-off `UPDATE stories SET word_index_at = NULL` migration), `status` (`generating`/`complete`/`failed`, NOT NULL DEFAULT `complete`), `error_message` TEXT nullable (set when `status='failed'`), `created_at`. RLS-scoped. Stories list and `getReadStoryContents` filter to `status='complete'`; `user_underused_kanji` is naturally limited to completed rows because it filters `read_count > 0` and only completed stories are markable as read. In-flight / failed rows are invisible everywhere except the Generator page's loading state.
- `profiles` ‚Äî auto-created on signup by the `handle_new_user()` trigger. Stores `display_name`, `openrouter_api_key_secret_id` (UUID reference into Supabase Vault ‚Äî the actual key is encrypted, never stored in plaintext on `profiles`), and `preferences` JSONB (defaults `{}`). The `preferences` shape is `{ generator: { model, formality, contentType, unseenWordTarget }, stories: { readFilter, sortMode, sortDir }, reader: { furigana, highlight } }` ‚Äî `generator.*` hydrates the Generator form (saved on every Generate; topic/style stay ephemeral), `stories.*` hydrates the Compositions page filters (saved on every filter change), `reader.*` hydrates `StoryDisplay`'s furigana / highlight controls (each `off`/`unseen`/`all`, saved every time a control is toggled). `stories.sortDir` is `asc`/`desc` and only applies to the directional sort modes (`newest`/`lastRead`) ‚Äî the score sort ignores it. The `unseenWordTarget` key is labelled "Unseen common words" ‚Äî how many words from the top 50 most-frequent never-encountered words (`getTopUnseenWords`) to nudge the LLM into using naturally; the prompt is explicit that this is only a nudge and the LLM should still introduce plenty of other new vocabulary. It is one of `none`/`1-2`/`3-5`/`5-10`. Defaults live client-side as `?? fallback` expressions, not on the column. All `preferred_*` columns were consolidated into this JSONB in migration `20260513000000`; the unused `preferred_prioritize_rare_kanji` was dropped in the same migration.
- `word_lookups` ‚Äî per-user *tap* history (analytics layer). Columns: `id`, `user_id`, `story_id`, `start_offset`, `end_offset`, `surface` (text as it appeared), `headword` (deinflected JMdict lemma, or surface fallback), `reading` (top JMdict reading, nullable), `looked_up_at`, `lookup_count`. UNIQUE on `(user_id, story_id, start_offset, end_offset)` ‚Äî re-tapping the same span upserts (refreshes `looked_up_at`, increments `lookup_count`). Indexed on `(user_id, headword, looked_up_at DESC)`. RLS-scoped. The carousel is no longer driven by this table; `get_word_usages` LEFT JOINs it so cards can surface "tapped N times" without making prior taps a gating signal. Both FKs cascade on delete.
- `story_word_occurrences` ‚Äî per-user word index. One row per regrouped span in a tokenized story: `id`, `user_id`, `story_id`, `start_offset`, `end_offset`, `surface`, `headword`, `reading`, `entry_id` (INTEGER nullable ‚Äî the JMdict id of the `WordResult` the indexer chose for this span, so the popover can hoist that exact entry instead of re-picking from JMdict's natural homophone ordering; null on legacy rows that haven't been re-indexed yet), `manual` (BOOLEAN NOT NULL DEFAULT FALSE ‚Äî TRUE iff the row was placed by the user via `set_story_word_overrides`). UNIQUE on `(story_id, start_offset, end_offset)`; indexed on `(user_id, headword)` for the carousel query. Populated by `index_story_words` (which now deletes only the story's *non-manual* rows before re-inserting, and filters out any algorithm span whose half-open `[start, end)` overlaps an existing manual row ‚Äî so manual rows survive every re-index and always shadow the algorithm's choice in their region). `set_story_word_overrides` does the region-replacement write that places manual rows. RLS-scoped. Both FKs cascade on delete, so deleting a story clears its occurrence rows.

RLS policies: `kanji` readable by authenticated users; `user_kanji`/`stories`/`profiles`/`word_lookups`/`story_word_occurrences` scoped to `auth.uid()`.

## Key RPCs

- `user_underused_kanji(p_limit INT DEFAULT 20)` ‚Äî returns the caller's under-exposed *seen* kanji (kanji that appear in at least one of the caller's read stories, ordered by exposure ASC, grade DESC, character ‚Äî deterministic, no random tiebreaker). Each story's character counts are multiplied by `read_count`, so re-reading a story increases exposure. Also returns `last_read_at` per kanji ‚Äî the `MAX(stories.last_read_at)` over the read stories containing it ‚Äî which powers the Stats Browse "last read" sort. Used by `getKanjiExposures` (called with a large limit) to derive `KanjiContext.seenKanji` and the kanji subtotal of the header score; no longer feeds the generation prompt (the prompt-injection of under-exposed kanji was removed). No longer joins `user_kanji.known` ‚Äî exposure is the only signal.
- `mark_story_read(p_story_id BIGINT)` / `undo_story_read(p_story_id BIGINT)` ‚Äî user-callable. `mark_‚Ä¶` increments `read_count` and refreshes `last_read_at` (setting `first_read_at` on the 0‚Üí1 transition). `undo_‚Ä¶` decrements with a floor of 0; the StoryReadButton additionally tracks per-session increments so the undo affordance only undoes same-session marks ‚Äî past-session reads can't be cleared from the UI.
- `record_word_lookup(p_story_id, p_start, p_end, p_surface, p_headword, p_reading)` ‚Äî user-callable upsert into `word_lookups`. Increments `lookup_count` + refreshes `looked_up_at` on conflict; refreshes `headword`/`reading`/`surface` so a future deinflection-rule change re-canonicalises the row.
- `index_story_words(p_story_id BIGINT, p_occurrences JSONB, p_version INTEGER)` ‚Äî user-callable bulk replace of one story's *non-manual* rows in `story_word_occurrences`. Deletes existing non-manual rows for the story, inserts every `{start, end, surface, headword, reading, entryId}` element of the JSONB array (skipping any whose `[start, end)` overlaps an existing manual row ‚Äî so manual overrides always survive re-indexes), and stamps `stories.word_index_at = now()` + `word_index_version = p_version` on success. Returns the stamp timestamp so the client can update its local state.
- `set_story_word_overrides(p_story_id, p_region_start, p_region_end, p_overrides JSONB)` ‚Äî user-callable region-replacement write for the manual-override UI. Deletes every row (manual or not) whose span intersects `[region_start, region_end)`, inserts each `{start, end, surface, headword, reading, entryId}` element of the JSONB array with `manual=TRUE`, and nulls `stories.word_index_at` so the backfill re-runs and refills any algorithm rows whose previous spans got collateral-damaged by the deletion. The override rows themselves are preserved by the next `index_story_words` call.
- `clear_story_word_overrides(p_story_id, p_region_start, p_region_end)` ‚Äî user-callable. Drops only the manual rows whose span intersects the region (algorithm rows are untouched) and nulls `word_index_at` so the algorithm can refill the gap on the next pass.
- `clear_all_story_word_overrides(p_story_id)` ‚Äî user-callable. Drops every manual row for the story (the "reset overrides" affordance on StoryDetail) and nulls `word_index_at` so the algorithm re-fills from scratch.
- `update_story_content(p_story_id, p_content)` ‚Äî user-callable. Replaces `stories.content`, sets `translations='{}'::jsonb`, nulls `word_index_at`/`word_index_version`, and deletes every `word_lookups` + `story_word_occurrences` row for the story (manual rows included ‚Äî their offsets are now stale). The backfill re-indexes the story on its next pass. The client uses this when the LLM's generated text itself is broken in a way no boundary override can fix.
- `get_word_usages(p_headword TEXT)` ‚Äî returns every `story_word_occurrences` row for the caller with the given headword, joined with the story (title, content, created_at). LEFT JOINs `word_lookups` so `looked_up_at` / `lookup_count` come along when the span was previously tapped. Filters to `stories.status = 'complete' AND read_count > 0` (the indexer eagerly indexes every complete story; this RPC is what limits popover examples to read stories). Ordered by `stories.created_at DESC, start_offset ASC` (newest stories first, in-text order within each).
- `get_word_encounters(p_headword TEXT)` ‚Äî total read-count-weighted encounters of `headword` across the caller's read stories. Mirrors the kanji-exposure pattern: every read of a story containing the headword contributes its `read_count`, so re-reads count separately. Powers the "N encounters" tag in the popover header.
- `get_story_word_encounters(p_story_id BIGINT)` ‚Äî per-occurrence encounter counts for one story (one row per indexed span; spans not yet indexed are absent). Each row's `encounters` is the same totals query as `get_word_encounters` evaluated for that span's headword. Used by `StoryDisplay` to underline zero-encounter (new-to-the-user) words; the underline is tinted by the word's JPDB rarity tier so it matches the WordPopover frequency badge (names and pre-frequency-load fall back to the accent colour).
- `get_user_word_encounters()` ‚Äî returns `(headword, encounters, last_read_at)` aggregated across the caller's read stories (read_count-weighted, mirrors `get_word_encounters` but in one round-trip for every headword; `last_read_at` is the `MAX(stories.last_read_at)` over the read stories containing the headword). Used by `VocabContext` to power the vocab subtotal in the header score and the Stats Browse "last read" sort.
- `get_per_story_word_occurrences()` ‚Äî returns `(story_id, headword, occurrences)` for every complete story owned by the caller (raw within-story counts, NOT read_count-weighted, since the Compositions payout asks "if I read this once more"). Used by `Stories.tsx` to compute per-card vocab payout.
- `set_openrouter_api_key(key text)` / `clear_openrouter_api_key()` ‚Äî user-callable; manage the Vault secret tied to the caller's profile. `set_‚Ä¶` self-heals if a stored secret-id no longer resolves in `vault.decrypted_secrets` (e.g., after `npm run sync-prod`).
- `update_preferences(p_patch jsonb)` ‚Äî user-callable atomic shallow merge into `profiles.preferences`. Uses `preferences || p_patch` so the patch overwrites top-level sections ‚Äî callers must send a section (`generator` or `stories`) in full to avoid clobbering its other keys. Returns the new preferences JSONB.
- `get_openrouter_api_key_for_user(p_user_id uuid)` ‚Äî service-role RPC that decrypts the Vault secret. Called only by Edge Functions.
- `strip_ruby(t text)` / `clean_generated_text(t text)` ‚Äî text helpers used internally by `user_underused_kanji` and migrations.

## Key Details

- Kanji grades use kanjiapi.dev convention: 1-6 for elementary, 8 for secondary (no grade 7).
- JLPT levels: 5 = easiest, 1 = hardest. Some kanji are unclassified (NULL).
- LLM output uses **Aozora ruby** notation: `kanji„Ääreading„Äã`. The prompt instructs character-level (not word-level) rubies, and tells the model to keep its kanji within three groups (the allowed list, kanji carried by the prioritized unseen common words, and kanji the topic/style genuinely needs) while writing each word with its ordinary modern spelling. `lib/furigana.ts` parses this on the client.
- Story generation runs as a **background task** in the Edge Function (via `EdgeRuntime.waitUntil`). The client `POST`s once, gets back `{ story_id }`, and polls `getStory(id)` every 3s until `status` flips. After 5 minutes without flipping, the client calls `markStoryFailed` to recover from a silently-killed worker. There is no streaming, no thinking-content display, and no validation ‚Äî whatever text the model returns is saved.
- Edge Functions read the OpenRouter API key from **Supabase Vault** via a service-role RPC (not from a plaintext profile column).
- OpenRouter API is OpenAI-compatible (`/v1/chat/completions`). Story generation enforces an `ALLOWED_MODELS` allow-list (see `generate-story/index.ts:9` for the current pin); translate-sentence pins its own model with a per-request token cap (see `translate-sentence/index.ts`).
- All local-dev env vars live in a single project-root `.env.local` (gitignored ‚Äî see `.env.local.example`). Vite reads it via `envDir` set in `client/vite.config.ts`; `supabase functions serve --env-file .env.local` loads it for Edge Functions. Only `VITE_*` vars are exposed to the browser bundle. Vars: `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` (client ‚Üí Supabase). Edge Functions also see `SUPABASE_URL` + `SUPABASE_SERVICE_ROLE_KEY` from the Deno env automatically.
- Local dev user: `supabase start` and `db reset` apply both `supabase/seed.sql` (kanji reference data) and `supabase/seed_dev.sql` (test account `dev@local.test` / `devpassword`, grade 1‚Äì3 kanji marked known, sample stories). The OpenRouter key is not seeded ‚Äî log in and paste it in Settings (it goes through the existing `set_openrouter_api_key()` RPC into Vault).

- Utilities & hooks: `camelCase.ts`. Tests are `<name>.test.ts` colocated next to the file under test.
