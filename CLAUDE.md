# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Is

A web app that generates short Japanese stories constrained to the kanji a user has previously seen (read in another story) plus a JLPT-N5 baseline, using OpenRouter LLM APIs. Designed for Japanese reading practice via comprehensible input. Each user has their own kanji exposure history, story history, and per-word LLM Q&A threads.

## Commands

```bash
# Install dependencies
npm install

# Start local Supabase (requires Docker)
npx supabase start

# Apply migrations and seed data (kanji + dev test user)
npx supabase db reset

# Generate seed SQL from data/kanji.json (only needed if kanji data changes)
npm run generate-seed

# Convert a JPDB Yomitan freq dict zip → client/public/frequency/jpdb.json
# (only needed when bumping the dict). Source: github.com/Kuuuube/yomitan-dictionaries
npm run generate-frequency -- path/to/JPDB_v2.2_Frequency_YYYY-MM-DD.zip

# Join the surface-keyed JPDB index with jmdict-simplified to produce
# client/public/frequency/jpdb-by-entry.json (per-JMdict-entry rank). Run
# after `generate-frequency` whenever either JPDB or JMdict is bumped.
# Source: github.com/scriptin/jmdict-simplified — pass the jmdict-eng-*.json.tgz
# from the latest release.
npm run generate-frequency-by-entry -- path/to/jmdict-eng-X.Y.Z+yyyymmdd.json.tgz

# Sync prod Supabase data into local instance
npm run sync-prod

# Generate TypeScript types from Supabase schema
npx supabase gen types typescript --local > client/src/lib/database.types.ts

# Run client dev server
npm run dev

# Build client (typecheck + Vite build)
npm run build

# Lint client
npm run lint --workspace=client

# Test client (Vitest)
npm test                              # one-shot
npm run test:watch --workspace=client # watch mode

# Serve Edge Functions locally
npx supabase functions serve --env-file .env.local

# Deploy Edge Functions
npx supabase functions deploy generate-story
npx supabase functions deploy translate-sentence
npx supabase functions deploy openrouter-usage
```

## Architecture

This is an npm workspaces monorepo. The root `package.json` declares `client` as the only workspace; there is no `server/` directory — the backend is entirely Supabase (Postgres + Edge Functions).

### Client (`client/`) — Vite + React + TypeScript

Talks directly to Supabase (DB via SDK, Edge Functions via `functions.invoke`). Routing in `client/src/App.tsx` lazy-loads `/` (Generator), `/stories`, `/stories/:id`, `/stats`, `/settings`; everything except `/login` is wrapped in `<ProtectedRoute>` + `<AppLayout>`. Plain CSS colocated as `Component.css` next to `Component.tsx` — no Tailwind, no CSS modules. Floating UI for popover positioning; `react-router-dom` for routing.

- **`lib/`** — pure utilities, no React imports. Notable: `generation.ts` (`buildPrompt()` only — difficulty is computed server-side now), `furigana.ts` (parses Aozora `kanji《reading》` ruby blocks), `applyOccurrences.ts` (`applyOccurrences(paragraphs, occurrences, ...)` folds `story_word_occurrences` rows into the char-level segmentation so the tap-target tree matches what's in the index — used by StoryDisplay once occurrences load. Manual override rows naturally take the place of the algorithm rows they replaced because the indexer treats them as just regular occurrence rows after `set_story_word_overrides` wrote them), `storySegments.ts` (walks `(cleanText, annotations)` into paragraph→sentence→part structure with character-level offsets), `sentenceSnippet.ts` (`extractSentenceSnippet` finds the sentence containing a span and rebases its annotations + highlight offsets, plus returns the sentence's absolute `sentenceStart` / `sentenceEnd` offsets in the source cleanText — used by the WordPopover to render each prior usage in context and to key the translation cache), `tokenizer.ts` (thin lazy-init wrapper around `@aiktb/kuromoji` exposing `tokenizeText` for boundary alignment + `posHintAtOffset(text, offset)` returning the POS of the kuromoji token starting at `offset`, backed by a small per-`text` result cache so the popover can reuse the regroup pass's tokenisation; dict files live in `client/public/dict/` and are copied there by the root `postinstall`), `regroupWords.ts` (async hybrid: kuromoji supplies token boundaries, JMdict (`lookupAtBoundary`) checks each candidate span — only matches that align with a kuromoji boundary or the run end are accepted, which rejects greedy false positives like 「があ」 while still letting deinflection merge across kuromoji tokens for spans like 「あります」; `StoryDisplay` falls back to char-level until both dict + tokenizer resolve), `storyWordIndex.ts` (`extractWordOccurrences(story)` runs the same regroup pass and emits every JMdict-matched span as `{start, end, surface, headword, reading}` — single-kanji CharParts are looked up, pure-kana CharParts are skipped; used by `WordIndexBackfillContext` to populate the server-side word index), `dictionary.ts` (JPDict IndexedDB lookup), `headword.ts` (`headwordFromHit` derives the canonical JMdict lemma from a `LookupHit` — uses `hit.base` for deinflected hits, otherwise the first non-`sK` primary `k` entry, falling back to `r[0]` when every kanji form is search-only or the entry is kana-only; returns null for 1-char no-match fallbacks so they're skipped from lookup history. The `sK` filter is what keeps the の particle's entry (k=[乃, 之], both `sK`) from stamping `乃` as canonical), `rarity.ts` (kanji-exposure scoring: saturating exponential up to count 10, then **hard-capped at the curve's value at 10** (~9.43) — further encounters of the same kanji contribute nothing. `totalScore` powers the header kanji subtotal, `readingScoreDelta` powers the kanji half of the "+X" tag on Compositions, `formatScore` is the display formatter — sub-1 values render as `0`, otherwise rounded integers. The shared `rawScore` curve is exported so `vocabScore.ts` can reuse the same shape), `vocabScore.ts` (per-headword analog of `rarity.ts`: `wordScore(count, rank) = rawScore(count) * frequencyWeight(rank) * VOCAB_SCALE`. `frequencyWeight` is a **sigmoid in rank** — `floor + (peak - floor) / (1 + (rank / MID_RANK)^K)` with peak 4.0, floor 0.15, MID_RANK 10,000, K=2 — so the top several thousand ranks all sit on a "core vocabulary" plateau near 4.0, the half-weight point is rank 10,000, and the curve asymptotes at the floor for very rare / unranked headwords (peak:floor ratio ≈ 27×). `VOCAB_SCALE` is `1/2.5` — a global dial that pulls the vocab subtotal back so the (much larger) headword pool doesn't drown out the kanji subtotal in the header. `totalVocabScore` and `vocabScoreDelta` take a `getRank: (headword) => number | null` callback (supplied by `VocabContext.getWordRank`, which calls `lookupFrequencyByCanonicalSync(headword)` first and falls back to `lookupFrequencySync(headword, null)` for stamps not covered by the by-entry index). Unranked headwords (null) collapse directly to the floor), `frequency.ts` (lazy-loaded JPDB rank lookups; fetches **two** indices on first use and caches them. (1) `/frequency/jpdb.json` is the surface-keyed YomiTan export — `lookupFrequency(headword, reading)` and `lookupBestFrequency(headwords[], reading)` resolve against it as a fallback for callers without a JMdict entry id; ranks > 100,000 are dropped at build time and resolve to the `very-rare` tier with `rank: null`. (2) `/frequency/jpdb-by-entry.json` is keyed by JMdict entry id (generated by `npm run generate-frequency-by-entry` joining jmdict-simplified with the surface index). Each record is `{rank, headword, reading, canonical}` where `headword/reading` is the variant whose rank won inside that one entry (so 暗い's entry stays 暗い rank 881 instead of letting the unrelated 位/「approximately」 entry's くらい rank 189 steal in), and `canonical` is the surface the indexer stamps on word_lookups / story_word_occurrences — the same precedence `headwordFromHit` uses (first non-`sK` kanji form, else `r[0]`), so a stamp can round-trip back to its entry via `lookupFrequencyByCanonicalSync`. The build script honours JMdict's `uk` misc tag — kana variants only contribute rank to an entry the dictionary marks as "usually kana" (貴方 → あなた rank 121; ありがとう wins as kana-only). It also filters `sK` ("search-only kanji") forms entirely: their JPDB ranks almost always belong to a different lexeme (の's entry has 乃/之 both `sK`, with the (乃, の) rank 13652 unrelated to the rank-1 particle), and JMdict explicitly says `sK` forms must never display. `lookupFrequencyByEntry(entryId)` / `lookupFrequencyByEntrySync` is the right call when you have a `WordResult` in hand (WordPopover taps + headword mode pass `hit.results[0].id`). `lookupFrequencyByCanonicalSync(canonical)` resolves a stamp surface back to its entry's rank via a `Map<canonical, entryId>` built on first load — collisions on the canonical surface (~3% of ranked entries, e.g. 〇 spans まる/れい/ゼロ) resolve to the lowest-rank entry. VocabContext uses canonical-keyed lookup so scoring captures the entry-resolved rank (貴方 stamps now weight as rank 121 instead of 3,151). `rankToTier` buckets ranks into `very-common`/`common`/`uncommon`/`rare`/`very-rare` and `TIER_LABEL` maps to display strings — used by the popover's frequency badge only; scoring no longer consumes tiers. `getFrequencyEntriesSync()` returns the full index flattened to one `{headword, reading, rank}` entry per headword (lowest-rank reading wins), sorted ascending — built and cached on first call; used by the Stats Browse section to paginate the top-N vocab), and the word-at-cursor stack (`lookupAtCursor`, `japaneseDeinflect`, `japaneseTransforms`, `languageTransformer`). `lookupAtBoundary`/`lookupExactSpan` take an optional kuromoji `posHint`: when it equals `動詞` and the exact match has no modern verb POS (archaic / obsolete senses are skipped, so the classical copula 也 doesn't block this), deinflection is run first and a verb result preempts the exact match — this fixes continuative forms like 「赤くなり、」 → なる, 「電車に乗り、」 → 乗る, and the rest of the 〜り/〜き/〜み/〜い endings before commas. Applies to both pure-kana and mixed-script surfaces; the only gate is the POS hint plus the absence of a modern verb POS. The regrouper and `storyWordIndex` derive the hint from their own tokenisation; `WordPopover` calls `posHintAtOffset` (which hits the tokeniser cache). `database.types.ts` is auto-generated — regenerate after migrations.
- **`api/client.ts`** — single boundary for all data ops: kanji (`getJlptN5Kanji` is the always-allowed baseline merged with the caller's `seenKanji` to form the prompt's allowed set; `getKanjiExposures` returns the read-count-weighted exposure map that `KanjiContext` derives `seenKanji` from; `getAllKanji` paginates the full joyo table for the Stats Browse section and caches it in memory), story CRUD, Edge Function invocations (`startStoryGeneration`/`translateSentence`), profile + Vault-backed API key management, lookup history (`recordWordLookup` upserts a `word_lookups` row on every popover open with a meaningful match — best-effort, errors are swallowed), word index (`indexStoryWords(storyId, occurrences)` bulk-replaces the story's non-manual rows in `story_word_occurrences` and stamps `stories.word_index_at`; called by the background `WordIndexBackfillContext`. `getStoryOccurrences(storyId)` reads every row including the `manual` flag — used by StoryDisplay to render tap targets from the index. `setStoryWordOverrides(storyId, regionStart, regionEnd, overrides)` does the region-replacement write for the manual-override UI, deleting every row overlapping the region and inserting the supplied overrides with `manual=TRUE`. `clearStoryWordOverrides(storyId, regionStart, regionEnd)` drops only the manual rows so the algorithm can refill. `updateStoryContent(storyId, content)` replaces the story body and wipes every offset-keyed cache — translations, word_lookups, occurrences — and nulls word_index_at so the backfill re-runs.), and `getWordUsages(headword)` returns every occurrence of the headword in the user's *read* stories joined with story content, plus optional `looked_up_at` / `lookup_count` from the `word_lookups` left-join. The "only show me read stories" filter lives in `get_word_usages` (SQL-side) so the indexer can run eagerly on every complete story without leaking unread examples into the popover. `getStories`/`getReadStoryContents` filter to `status='complete'` so in-flight rows never reach the Stories list. `getInFlightGeneration` returns the user's most recent non-complete row (used by `GenerationContext` to hydrate); `markStoryFailed` flips a stale row when the Edge Function dies silently. Snake↔camel conversion happens here.
- **`contexts/`** — `AuthContext` (session/profile), `GenerationContext` (background-generation state machine — calls `startStoryGeneration`, then polls `getStory(id)` every 3s until `status` flips to `complete`/`failed`; on mount, hydrates from any `generating` row so reloading mid-generation resumes the loading state. Failed rows are best-effort deleted before the next `generate()` retry so they don't accumulate. The completed story is intentionally **not** surfaced through this context — the Generator page is fire-and-forget, and finished stories are only viewable on the Compositions / Story Detail pages), `KanjiContext` (per-kanji read-count-weighted exposure map; `seenKanji` is derived as `new Set(kanjiExposures.keys())` — the "known" concept was dropped, a kanji is just considered seen iff the user has read a story containing it. `refreshKanjiExposures` is called by `StoryReadButton` so marking a story read updates the header total live), `VocabContext` (per-headword read-count-weighted encounter counts; loads once via `getUserWordEncounters` — which paginates through PostgREST's `db-max-rows` cap in 1000-row chunks — and exposes a `Map<headword, number>` plus a `getWordRank` callback. `refreshVocabEncounters` awaits **both** the encounter RPC and `loadFrequencyIndex()` before flipping `vocabEncountersLoaded`, since vocab scoring is frequency-weighted and a partial load would visibly jump the header total. `AppLayout` hides the entire nav score until both `kanjiExposuresLoaded` and `vocabEncountersLoaded` are true; `Stats` blocks the whole page on the same combined gate. `refreshVocabEncounters` is called by `StoryReadButton` alongside the kanji refresh so the score updates live on read), `DictionaryContext` (lookup cache), `WordIndexBackfillContext` (background queue for indexing every complete unstamped story — read state is *not* a gate, so popover taps on a freshly-generated story are instant once it's marked read. Hydrates from `getStoriesNeedingIndex` once auth+dict are ready, then runs `extractWordOccurrences` + `indexStoryWords` serially with a 200ms gap, exposing `remaining`/`processing`/`currentStoryId` for the Settings panel. `GenerationContext` calls `refresh()` when a story flips to `complete` so newly-generated rows enter the queue without waiting for next session. Always-on: no pause control.), `StoriesContext` (lives in `AppLayout` so the cached Compositions list survives route changes — the page used to refetch `getStories` + `getPerStoryWordOccurrences` on every tab switch. Loads both once on auth-ready; exposes `addStory` (called by `GenerationContext` when a row flips to `complete`), `removeStory` (called after `deleteStory` in `Stories.tsx` and `StoryDetail.tsx`, and by `GenerationContext` when cleaning up a previous failed row), and `applyStoryUpdate` (called from `StoryDetail`'s `StoryReadButton.onChange` so `read_count` / `last_read_at` updates propagate to the cached list). When `WordIndexBackfillContext.processing` transitions from true back to false, only the per-story occurrences are re-fetched so freshly-indexed stories' payout tags update without forcing a full list refetch.).
- **`components/`** — notable behaviors: `StoryDisplay` renders `<ruby>` annotations + tap-to-translate; once `story_word_occurrences` is loaded for the story (after the first index pass) it switches from the local regroup pass to `applyOccurrences` so tap targets line up with the carousel cards and manual override rows are visible. Each tap passes the occurrence's stored `headword` to the popover via `lookupHeadword` so the dictionary lookup uses the canonical (or override) lemma instead of the raw surface — that's how a manual row like `{start: 0, end: 3, surface: 「野さい」, headword: 野菜}` makes the popover show 野菜's entry even though JMdict has nothing for 「野さい」. The occurrence's `entry_id` rides along via `lookupEntryId`; the popover hoists the matching `WordResult` to position 0 in `hit.results` so `headwordFromHit` picks the JMdict entry the indexer actually chose. Without this, the popover's redo-lookup (which has no POS context) lets JMdict's natural ordering surface the wrong homophone — ふっても → フル (loanword "full") instead of 降る, いきます → 幾 (prefix) instead of 行く. The "Override match" affordance on the popover puts the page into `StoryOverrideEditor` (region select + per-character split toggles + JMdict candidate picker per sub-span); saving routes through `setStoryWordOverrides` and bumps an internal nonce so occurrences re-fetch immediately. `StoryDetail`'s pencil button opens an inline textarea over `stories.content` and calls `updateStoryContent` on save — used when the LLM produced a broken spelling that no boundary edit can fix (e.g. 野さい → 野菜《やさい》). The ↻ button next to the pencil clears every manual override for the story via `clearAllStoryWordOverrides` (the "reset overrides" path). Any action that nulls `word_index_at` on the rendered story — override save, content edit, reset-overrides — flips a `regenerating` flag in `StoryDetail` that swaps the story body for the "Preparing story" loader. `StoryDetail` watches `WordIndexBackfillContext.currentStoryId`; when it transitions away from `story.id` the parent refetches the story (so `word_index_at` is restored locally — otherwise popover taps would stay disabled until a page reload) and clears the flag. A 15s fallback timer drops the flag if the backfill never picks the story up. The `regenerating` flag flows down as a prop, and StoryDisplay's own override save calls an `onRegenerationStart` callback to lift the flag synchronously before the DB write. `BrowseSection` renders the Stats Browse widget (mode toggle Kanji/Vocab, "Seen only" filter, kanji-mode JLPT + grade multi-select chips, vocab-mode 100-rank window slider over the top 50,000 JPDB headwords; tap a kanji card → opens `KanjiInlineDetail` in a floating-ui overlay modal; tap a vocab card → opens `WordPopover` in `headword` mode); `WordPopover` shows the tapped word's dictionary entry plus an AI translation of the *sentence* containing the tap. The popover is structured as a sticky header (headword + JPDB frequency tier badge with rank subscript via `lookupFrequencyByEntry(hit.results[0].id)`, falling back to a candidate-list `lookupBestFrequency` only for 1-char no-match hits with no JMdict entry to key on + senses + headword's kanji chips — identical across all cards, so visual repetition is avoided) over a horizontal carousel of "usage" cards. Tapping a kanji chip in the sticky header swaps the popover body for `KanjiInlineDetail` (meanings + on/kun readings + grade + JLPT, view-only). Card 0 is the current tap; cards 1..N are every other occurrence of the same headword across the user's tokenized stories (fetched via `getWordUsages` from `story_word_occurrences`). Each card carries its own `(storyId, start, end)`. Below the sentence snippet, the card shows an English translation of that sentence with a "↻ Regenerate" button — translations are fetched lazily on first view via `translateSentence`, cached server-side in `stories.translations` keyed by the sentence's char offsets, and cached client-side: the current story's translations come from props (`StoryDisplay` owns the map and `onTranslationUpdated` bubbles updates), other stories' translations are kept in a popover-local `otherStoryTranslations` map so swiping back doesn't refetch. Tapping a different word in the same sentence shows the same cached translation instantly. Navigation: arrow buttons + ←/→ keys + horizontal touch swipe + position indicator. Hidden when there's only a current card. The current span is filtered out of `usages` to avoid duplicating it as both card 0 and card N. `StoryReadButton` only handles read-state — word indexing is the backfill context's job and runs ahead of the first read.
- **`types/index.ts`** — shared interfaces (`Kanji`, `Story`, `SentenceTranslation`, `StoryTranslations`, `WordUsage`, `Profile`).
- **Tests** — Vitest, colocated `*.test.ts`. Pure-lib only — no React rendering tests. `src/test/setup.ts` stubs Vite env vars so `lib/supabase.ts` is importable transitively.

### Supabase (`supabase/`)

- `config.toml` — local CLI config; `db.seed.sql_paths` runs both `seed.sql` and `seed_dev.sql` on `db reset`.
- `seed.sql` — joyo kanji reference data (regenerated from `data/kanji.json` via `npm run generate-seed`).
- `seed_dev.sql` — dev test user (`dev@local.test` / `devpassword`) and sample stories. Idempotent on re-runs. Still inserts grade 1–3 rows into `user_kanji` for historical parity, but those rows are inert — no code path reads `user_kanji.known` anymore.
- `migrations/` — timestamped SQL files (`YYYYMMDDHHMMSS_*.sql`). Run `ls supabase/migrations/` for current state.
- `functions/_shared/` — utilities shared by Edge Functions (`openrouter.ts`, `story.ts`, `text.ts`).
- `functions/generate-story/` — runs as a **background task**. Auths user, rejects (409) if a `status='generating'` row already exists for the user, inserts a placeholder `stories` row with `status='generating'`, returns 202 + `{ story_id }` immediately, then uses `EdgeRuntime.waitUntil` to call OpenRouter, parse the title/content, compute difficulty (loads kanji metadata server-side), and `UPDATE` the row to `status='complete'` (or `status='failed'` + `error_message` on any error). The client never streams from this function — it polls `stories.id` until status flips. Pins to a single model via `ALLOWED_MODELS` allow-list (see `generate-story/index.ts:24`). Maps 401/402/429 to user-friendly errors.
- `functions/translate-sentence/` — translates a single sentence within a story to natural English. Input is `{ story_id, sentence_start, sentence_end, regenerate? }` — sentence offsets are char offsets in the *cleaned* content (ruby-stripped) as computed by the client's `extractSentenceSnippet`, so both sides agree on sentence bounds. The prompt sends the **full cleaned passage** alongside the target sentence so the model can resolve dropped subjects, pronouns, names, and tone consistency — Japanese drops a lot, and a context-free translation often guesses wrong. On cache hit (and `regenerate=false`) returns the stored translation without calling OpenRouter; otherwise calls the model, upserts `stories.translations[<start>-<end>] = { text, model, generated_at }`, and returns the translation. Pins a single model (see `translate-sentence/index.ts`).
- `functions/openrouter-usage/` — surfaces OpenRouter credit/usage to the client.

## Data Model

- `kanji` — reference data (read-only). `character` PK, `grade` (1-6, 8=secondary), `jlpt` (5=easiest, 1=hardest, NULL allowed), `meanings`, `readings_on`, `readings_kun`
- `user_kanji` — legacy per-user known state. Kept in the schema for safety but no longer read or written by any client code or RPC. "Seen kanji" is now derived purely from read stories (see `KanjiContext` / `getKanjiExposures`); a future migration may drop this table.
- `stories` — per-user stories. Columns include `title`, `content`, `paragraphs`, `topic`, `formality` (`impolite`/`casual`/`polite`/`keigo`), `content_type` (`fiction`/`nonfiction`), `filters` JSONB (legacy, no longer written by the client), `allowed_kanji` (TEXT — concatenated kanji string, not JSON), `difficulty` JSONB, `translations` JSONB nullable (`{ "<sentence_start>-<sentence_end>": { text, model, generated_at } }` — a sentence-level cache populated by the `translate-sentence` Edge Function), `read_count` (INT NOT NULL, defaults 0; weights kanji-exposure aggregation so re-reads count), `first_read_at` / `last_read_at`, `word_index_at` TIMESTAMPTZ nullable (stamped by `index_story_words` when the story's regrouped spans are bulk-uploaded into `story_word_occurrences`; null means the indexer hasn't run yet and the next backfill pass will trigger it), `word_index_version` INTEGER nullable (stamped alongside `word_index_at`; the backfill treats rows whose stamped version is below `WORD_INDEX_VERSION` in `lib/storyWordIndex.ts` as out-of-date and re-indexes them — bumping the constant in code replaces the one-off "null out everything" migrations we used to ship when regroup/deinflect rules changed), `status` (`generating`/`complete`/`failed`, NOT NULL DEFAULT `complete`), `error_message` TEXT nullable (set when `status='failed'`), `created_at`. RLS-scoped. Stories list and `getReadStoryContents` filter to `status='complete'`; `user_underused_kanji` is naturally limited to completed rows because it filters `read_count > 0` and only completed stories are markable as read. In-flight / failed rows are invisible everywhere except the Generator page's loading state.
- `profiles` — auto-created on signup by the `handle_new_user()` trigger. Stores `display_name`, `openrouter_api_key_secret_id` (UUID reference into Supabase Vault — the actual key is encrypted, never stored in plaintext on `profiles`), and `preferences` JSONB (defaults `{}`). The `preferences` shape is `{ generator: { model, formality, paragraphs, contentType, unknownKanjiTarget }, stories: { readFilter, paragraphFilter, sortMode } }` — `generator.*` hydrates the Generator form (saved on every Generate; topic/style stay ephemeral), `stories.*` hydrates the Compositions page filters (saved on every filter change). The `unknownKanjiTarget` key is labelled "Unseen kanji" in the UI — the meaning is "how many kanji NOT in the allowed list to inject as stretch content". Defaults live client-side as `?? fallback` expressions, not on the column. All `preferred_*` columns were consolidated into this JSONB in migration `20260513000000`; the unused `preferred_prioritize_rare_kanji` was dropped in the same migration.
- `word_lookups` — per-user *tap* history (analytics layer). Columns: `id`, `user_id`, `story_id`, `start_offset`, `end_offset`, `surface` (text as it appeared), `headword` (deinflected JMdict lemma, or surface fallback), `reading` (top JMdict reading, nullable), `looked_up_at`, `lookup_count`. UNIQUE on `(user_id, story_id, start_offset, end_offset)` — re-tapping the same span upserts (refreshes `looked_up_at`, increments `lookup_count`). Indexed on `(user_id, headword, looked_up_at DESC)`. RLS-scoped. The carousel is no longer driven by this table; `get_word_usages` LEFT JOINs it so cards can surface "tapped N times" without making prior taps a gating signal. Both FKs cascade on delete.
- `story_word_occurrences` — per-user word index. One row per regrouped span in a tokenized story: `id`, `user_id`, `story_id`, `start_offset`, `end_offset`, `surface`, `headword`, `reading`, `entry_id` (INTEGER nullable — the JMdict id of the `WordResult` the indexer chose for this span, so the popover can hoist that exact entry instead of re-picking from JMdict's natural homophone ordering; null on legacy rows that haven't been re-indexed yet), `manual` (BOOLEAN NOT NULL DEFAULT FALSE — TRUE iff the row was placed by the user via `set_story_word_overrides`). UNIQUE on `(story_id, start_offset, end_offset)`; indexed on `(user_id, headword)` for the carousel query. Populated by `index_story_words` (which now deletes only the story's *non-manual* rows before re-inserting, and filters out any algorithm span whose half-open `[start, end)` overlaps an existing manual row — so manual rows survive every re-index and always shadow the algorithm's choice in their region). `set_story_word_overrides` does the region-replacement write that places manual rows. RLS-scoped. Both FKs cascade on delete, so deleting a story clears its occurrence rows.

RLS policies: `kanji` readable by authenticated users; `user_kanji`/`stories`/`profiles`/`word_lookups`/`story_word_occurrences` scoped to `auth.uid()`.

## Key RPCs

- `user_underused_kanji(p_limit INT DEFAULT 20)` — returns the caller's under-exposed *seen* kanji (kanji that appear in at least one of the caller's read stories, ordered by exposure ASC, grade DESC, character — deterministic, no random tiebreaker). Each story's character counts are multiplied by `read_count`, so re-reading a story increases exposure. Used by `getKanjiExposures` (called with a large limit) to derive `KanjiContext.seenKanji` and the kanji subtotal of the header score; no longer feeds the generation prompt (the prompt-injection of under-exposed kanji was removed). No longer joins `user_kanji.known` — exposure is the only signal.
- `mark_story_read(p_story_id BIGINT)` / `undo_story_read(p_story_id BIGINT)` — user-callable. `mark_…` increments `read_count` and refreshes `last_read_at` (setting `first_read_at` on the 0→1 transition). `undo_…` decrements with a floor of 0; the StoryReadButton additionally tracks per-session increments so the undo affordance only undoes same-session marks — past-session reads can't be cleared from the UI.
- `record_word_lookup(p_story_id, p_start, p_end, p_surface, p_headword, p_reading)` — user-callable upsert into `word_lookups`. Increments `lookup_count` + refreshes `looked_up_at` on conflict; refreshes `headword`/`reading`/`surface` so a future deinflection-rule change re-canonicalises the row.
- `index_story_words(p_story_id BIGINT, p_occurrences JSONB, p_version INTEGER)` — user-callable bulk replace of one story's *non-manual* rows in `story_word_occurrences`. Deletes existing non-manual rows for the story, inserts every `{start, end, surface, headword, reading, entryId}` element of the JSONB array (skipping any whose `[start, end)` overlaps an existing manual row — so manual overrides always survive re-indexes), and stamps `stories.word_index_at = now()` + `word_index_version = p_version` on success. Returns the stamp timestamp so the client can update its local state.
- `set_story_word_overrides(p_story_id, p_region_start, p_region_end, p_overrides JSONB)` — user-callable region-replacement write for the manual-override UI. Deletes every row (manual or not) whose span intersects `[region_start, region_end)`, inserts each `{start, end, surface, headword, reading, entryId}` element of the JSONB array with `manual=TRUE`, and nulls `stories.word_index_at` so the backfill re-runs and refills any algorithm rows whose previous spans got collateral-damaged by the deletion. The override rows themselves are preserved by the next `index_story_words` call.
- `clear_story_word_overrides(p_story_id, p_region_start, p_region_end)` — user-callable. Drops only the manual rows whose span intersects the region (algorithm rows are untouched) and nulls `word_index_at` so the algorithm can refill the gap on the next pass.
- `clear_all_story_word_overrides(p_story_id)` — user-callable. Drops every manual row for the story (the "reset overrides" affordance on StoryDetail) and nulls `word_index_at` so the algorithm re-fills from scratch.
- `update_story_content(p_story_id, p_content)` — user-callable. Replaces `stories.content`, sets `translations='{}'::jsonb`, nulls `word_index_at`/`word_index_version`, and deletes every `word_lookups` + `story_word_occurrences` row for the story (manual rows included — their offsets are now stale). The backfill re-indexes the story on its next pass. The client uses this when the LLM's generated text itself is broken in a way no boundary override can fix.
- `get_word_usages(p_headword TEXT)` — returns every `story_word_occurrences` row for the caller with the given headword, joined with the story (title, content, created_at). LEFT JOINs `word_lookups` so `looked_up_at` / `lookup_count` come along when the span was previously tapped. Filters to `stories.status = 'complete' AND read_count > 0` (the indexer eagerly indexes every complete story; this RPC is what limits popover examples to read stories). Ordered by `stories.created_at DESC, start_offset ASC` (newest stories first, in-text order within each).
- `get_word_encounters(p_headword TEXT)` — total read-count-weighted encounters of `headword` across the caller's read stories. Mirrors the kanji-exposure pattern: every read of a story containing the headword contributes its `read_count`, so re-reads count separately. Powers the "N encounters" tag in the popover header.
- `get_story_word_encounters(p_story_id BIGINT)` — per-occurrence encounter counts for one story (one row per indexed span; spans not yet indexed are absent). Each row's `encounters` is the same totals query as `get_word_encounters` evaluated for that span's headword. Used by `StoryDisplay` to apply the accent underline to zero-encounter (new-to-the-user) words.
- `get_user_word_encounters()` — returns `(headword, encounters)` aggregated across the caller's read stories (read_count-weighted, mirrors `get_word_encounters` but in one round-trip for every headword). Used by `VocabContext` to power the vocab subtotal in the header score.
- `get_per_story_word_occurrences()` — returns `(story_id, headword, occurrences)` for every complete story owned by the caller (raw within-story counts, NOT read_count-weighted, since the Compositions payout asks "if I read this once more"). Used by `Stories.tsx` to compute per-card vocab payout.
- `set_openrouter_api_key(key text)` / `clear_openrouter_api_key()` — user-callable; manage the Vault secret tied to the caller's profile. `set_…` self-heals if a stored secret-id no longer resolves in `vault.decrypted_secrets` (e.g., after `npm run sync-prod`).
- `update_preferences(p_patch jsonb)` — user-callable atomic shallow merge into `profiles.preferences`. Uses `preferences || p_patch` so the patch overwrites top-level sections — callers must send a section (`generator` or `stories`) in full to avoid clobbering its other keys. Returns the new preferences JSONB.
- `get_openrouter_api_key_for_user(p_user_id uuid)` — service-role RPC that decrypts the Vault secret. Called only by Edge Functions.
- `strip_ruby(t text)` / `clean_generated_text(t text)` — text helpers used internally by `user_underused_kanji` and migrations.

## Key Details

- Kanji grades use kanjiapi.dev convention: 1-6 for elementary, 8 for secondary (no grade 7).
- JLPT levels: 5 = easiest, 1 = hardest. Some kanji are unclassified (NULL).
- LLM output uses **Aozora ruby** notation: `kanji《reading》`. The prompt instructs character-level (not word-level) rubies, fall back to hiragana when a kanji is not in the allow-list. `lib/furigana.ts` parses this on the client.
- Story generation runs as a **background task** in the Edge Function (via `EdgeRuntime.waitUntil`). The client `POST`s once, gets back `{ story_id }`, and polls `getStory(id)` every 3s until `status` flips. After 3 minutes without flipping, the client calls `markStoryFailed` to recover from a silently-killed worker. There is no streaming, no thinking-content display, and no validation — whatever text the model returns is saved.
- Edge Functions read the OpenRouter API key from **Supabase Vault** via a service-role RPC (not from a plaintext profile column).
- OpenRouter API is OpenAI-compatible (`/v1/chat/completions`). Story generation enforces an `ALLOWED_MODELS` allow-list (see `generate-story/index.ts:9` for the current pin); translate-sentence pins its own model with a per-request token cap (see `translate-sentence/index.ts`).
- All local-dev env vars live in a single project-root `.env.local` (gitignored — see `.env.local.example`). Vite reads it via `envDir` set in `client/vite.config.ts`; `supabase functions serve --env-file .env.local` loads it for Edge Functions. Only `VITE_*` vars are exposed to the browser bundle. Vars: `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` (client → Supabase). Edge Functions also see `SUPABASE_URL` + `SUPABASE_SERVICE_ROLE_KEY` from the Deno env automatically.
- Local dev user: `supabase start` and `db reset` apply both `supabase/seed.sql` (kanji reference data) and `supabase/seed_dev.sql` (test account `dev@local.test` / `devpassword`, grade 1–3 kanji marked known, sample stories). The OpenRouter key is not seeded — log in and paste it in Settings (it goes through the existing `set_openrouter_api_key()` RPC into Vault).

## Conventions

- Components: `PascalCase.tsx` with optional colocated `.css` of the same name.
- Utilities & hooks: `camelCase.ts`. Tests are `<name>.test.ts` colocated next to the file under test.
- DB columns: `snake_case`; TypeScript fields: `camelCase`. The auto-generated `database.types.ts` exposes the snake_case shape — convert at the API boundary in `api/client.ts`.
- Pages compose contexts + components; `lib/` stays free of React imports.
- Don't reach into Supabase tables that have RLS from the client without an authenticated session — queries will silently return empty.
- When adding a migration, regenerate `client/src/lib/database.types.ts` and update `types/index.ts` if user-facing shapes change.
